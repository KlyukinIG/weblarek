# Интернет-магазин «Web-Larёk»
«Web-Larёk» — это интернет-магазин с товарами для веб-разработчиков, где пользователи могут просматривать товары, добавлять их в корзину и оформлять заказы. Сайт предоставляет удобный интерфейс с модальными окнами для просмотра деталей товаров, управления корзиной и выбора способа оплаты, обеспечивая полный цикл покупки с отправкой заказов на сервер.

## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и  отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component
Является базовым классом для всех компонентов интерфейса.
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`


#### Класс Api
Содержит в себе базовую логику отправки запросов.

Конструктор:  
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.

Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер  
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter
Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` -  хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.

#### Данные
В приложении используются две основные сущности, описывающие предметную область: товар и покупатель. Для них определены следующие интерфейсы:

`IProduct` — интерфейс, описывающий карточку товара, которая приходит с сервера или отображается в каталоге.

`id: string` — уникальный идентификатор товара.
`description: string` — подробное описание товара.
`image: string` — ссылка на изображение товара.
`title: string` — название товара.
`category: string` — категория, к которой относится товар.
`price: number | null` — цена товара. Может быть `null`, если товар бесценный (например, «бесценно»).

`IBuyer` — интерфейс, описывающий данные, которые пользователь вводит при оформлении заказа.
`payment: TPayment` — выбранный способ оплаты (например, 'online' или 'cash').
`email: string` — электронная почта покупателя.
`phone: string` — номер телефона покупателя.
`address: string` — адрес доставки.

`TValidationErrors`) — описывает структуру объекта с ошибками, возвращаемого методом `validate()` класса `Buyer`.

`TPayment` — тип описывает способы облаты товаров `online` или `cash`.

##### Моделях данных

Для хранения и управления состоянием приложения определены три класса-модели. 

#### Класс ProductCatalog
Отвечает за хранение данных о товарах, доступных в магазине, а также за товар, выбранный пользователем для просмотра в модальном окне.

Конструктор:  
`constructor(events: EventEmitter)` — принимает экземпляр брокера событий для возможности оповещения об изменениях.

Поля класса: 
`_items: IProduct[]` - хранит массив всех товаров;
`_preview: string | null` - хранит товар, выбранный для подробного отображения;

Методы класса:  
`setItems(items: IProduct[]): void` - сохранение массива товаров полученного в параметрах метода. Принимает на вход массив объектов, соответствующих интерфейсу `IProduct`;
`getItems(): IProduct[]` - получение массива товаров из модели;
`getProduct(id: string): IProduct | undefined` - получение одного товара по его id;
`setPreview(id: string): void` - сохранение товара для подробного отображения. Принимает строку с идентификатором товара.;
`getPreview(): IProduct | undefined` - получение товара для подробного отображения.

#### Класс ShoppingCart
Отвечает за хранение товаров, которые пользователь добавил в корзину, и предоставление данных о ней

Конструктор:  
`constructor(events: EventEmitter)` — принимает экземпляр брокера событий для возможности оповещения об изменениях.

Поля класса: 
`_items: IProduct[]` - хранит массив товаров, выбранных покупателем для покупки.

Методы класса:  
`getItems(): IProduct[]` - получение массива товаров, которые находятся в корзине;
`addItem(item: IProduct): void` - добавление товара, который был получен в параметре, в массив корзины;
`removeItem(itemId: string): void` - удаление товара, полученного в параметре из массива корзины;
`clear(): void` - очистка корзины;
`getTotalPrice(): number` - получение стоимости всех товаров в корзине;
`getCount(): number` - получение количества товаров в корзине;
`isInCart(itemId: string): boolean` - проверка наличия товара в корзине по его id, полученного в параметр метода.


#### Класс Buyer
Отвечает за хранение и валидацию персональных данных покупателя, введенных на этапах оформления заказа.

Конструктор:
`constructor(events: EventEmitter)` — принимает экземпляр брокера событий для возможности оповещения об изменениях.

Поля класса: 
`_payment: TPayment | null`- вид оплаты;
`_address: string` - адреc;
`_email: string` - телефон;
`_phone: string` - email.

Методы класса:  
`setField(field: keyof IBuyer, value: string): void` - универсальный метод для сохранения значения одного поля. Принимает название поля и его значение;
`getData(): IBuyer` - получение всех данных покупателя;
`clear(): void` - очистка данных покупателя;
`validate(): TValidationErrors` - выполняет валидацию данных покупателя. Возвращает объект типа `TValidationErrors`, содержащий сообщения об ошибках только для тех полей, которые не прошли валидацию. Если поле валидно, оно отсутствует в возвращаемом объекте.
 *Пример возвращаемого значения:* `{ payment: 'Выберите способ оплаты', email: 'Укажите email' }`

#### Инициализация приложения

1. **Создание брокера событий** `EventEmitter` - для обмена сообщениями между компонентами

2. **Создание экземпляров моделей данных**:
`ProductCatalog` - для управления каталогом товаров
`ShoppingCart` - для управления корзиной
`Buyer` - для хранения данных покупателя

3. **Создание экземпляров View-компонентов**:
`Gallery` - отображение каталога
`Modal` - управление модальными окнами
`Header` - шапка сайта со счётчиком корзины
`Basket` - корзина 
`OrderForm` - форма заказа
`ContactsForm` - форма контактов
`Success` - сообщение об успехе 
4. **Загрузка товаров с сервера** через `api.getProducts()`

5. **Регистрация обработчиков событий** - подписка на все события

#### Сценарии работы
##### 1. Отображение каталога
- При загрузке страницы вызывается `loadProducts()`, которая получает товары с сервера
- Товары сохраняются в модель `catalog` через `setItems()`
- Модель генерирует событие `catalog:changed`
- Презентер обрабатывает событие и вызывает `updateCatalog()`
- Для каждого товара создаётся карточка `CardCatalog` с колбэком `onClick`, который генерирует событие `card:select`
- Карточки отображаются в галерее через `gallery.catalog`

##### 2. Просмотр товара
- Пользователь кликает по карточке товара в каталоге
- Карточка генерирует событие `card:select` с `id` товара
- Презентер вызывает `catalog.setPreview(id)`
- Модель сохраняет выбранный товар и генерирует событие `preview:changed`
- Презентер вызывает `updatePreview()`, создаёт карточку `CardPreview` и открывает модальное окно

##### 3. Добавление/удаление товара из корзины
- Пользователь нажимает кнопку "Купить" или "Удалить" в превью
- Карточка генерирует событие `card:buy` с `id` товара
- Презентер проверяет наличие товара в корзине через `cart.isInCart()`
- Вызывается `cart.addItem()` или `cart.removeItem()`
- Модель корзины генерирует событие `basket:changed`
- Презентер обновляет отображение корзины через `updateBasket()` и обновляет счётчик в шапке

##### 4. Открытие корзины и оформление заказа
- Пользователь кликает на иконку корзины в шапке
- `Header` генерирует событие `basket:open`
- Презентер открывает модальное окно с компонентом корзины

##### 5. Заполнение формы заказа
- При выборе способа оплаты форма генерирует событие `order:paymentSelected`
- При вводе адреса форма генерирует событие `order:inputChanged`
- Презентер сохраняет данные в модель `buyer` через `setField()`
- Модель генерирует событие `order:changed`
- Презентер вызывает `updateOrderForm()`, которая обновляет отображение ошибок и состояние кнопки "Далее"

##### 6. Заполнение формы контактов
- Пользователь нажимает "Далее" в форме заказа
- Презентер проверяет валидность первого шага через `buyer.validate()`
- Если всё заполнено верно, открывается форма контактов
- При вводе email/телефона форма генерирует событие `contacts:inputChanged`
- Презентер сохраняет данные и обновляет отображение ошибок

##### 7. Отправка заказа
- Пользователь нажимает "Оплатить" в форме контактов
- Презентер проверяет валидность всех данных
- Формируется объект заказа и отправляется на сервер через `api.postOrder()`
- При успешном ответе показывается сообщение об успехе
- Корзина и данные покупателя очищаются
